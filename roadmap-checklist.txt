Remaining OWL/DL work that still fits the DAG direction
[x] `FORALL` / universal restriction
    - Implemented in `src/constraints.py`, `src/dag_eval.py`, and `src/ontology_parse.py`
    - Current semantics: min over neighbor scores, with vacuous truth when no outgoing edges exist for that property
[x] Negation / complement-like scoring
    - Implemented as a `NEGATION` DAG node in `src/constraints.py` and `src/dag_eval.py`
    - Current semantics: `score(not D) = 1 - score(D)`
[~] Disjointness / contradiction penalties
    - Compiler now supports direct `owl:disjointWith` by compiling it to a negated class constraint
    - Still missing: broader contradiction-handling machinery and richer disjointness patterns such as global conflict tracking
[x] Subclass and subproperty hierarchy propagation
    - Implemented as optional ABox materialization in `src/ontology_parse.py`
    - Current behavior: propagate `rdf:type` over `rdfs:subClassOf` and property assertions over `rdfs:subPropertyOf`
[~] Compilation of named OWL classes into DAGs
    - Implemented for a useful minimal fragment in `src/ontology_parse.py`
    - Supported now includes `owl:unionOf` compilation into native DAG `UNION` nodes
    - Still missing: broader class-expression coverage and cleaner normalization / compilation structure
[~] Parsing of `rdfs:domain` / `rdfs:range` expressions per property
    - Implemented in `src/ontology_parse.py` as preserved property-axiom expression parsing
    - Supported now: atomic classes, `owl:unionOf`, datatype ranges via `owl:onDatatype` + `owl:withRestrictions`, and other currently renderable expressions
    - Also implemented: opt-in query-time DAG augmentation, opt-in atomic domain/range materialization, and opt-in Horn-safe antecedent materialization for atomic/intersection targets
    - Still missing: broader ontology-style inference/materialization for complex non-atomic domain/range expressions
[~] Cardinality restrictions
    - Implemented natively for immediate-neighbor local semantics in `src/constraints.py`, `src/dag_eval.py`, and `src/ontology_parse.py`
    - Supported now: exact, min, and max cardinality; qualified and unqualified; forward and backward traversal; strict top-k aggregation mode
    - Current semantics: graph-local top-k scoring over observed neighbors, not open-world OWL cardinality semantics
    - Deferred for later: fuzzy cardinality scoring / delta-based spread tuning
    - Still missing: any decision about oracle-comparison treatment for max/exact cases
[~] Inverse properties
    - Implemented via targeted role saturation in `src/ontology_parse.py`
    - Also implemented natively for inverse property expressions on restrictions via backward DAG traversal in `src/dag_eval.py`
    - Current scope: reusable inverse consequences still flow through the targeted saturation path, while direct inverse query semantics can avoid materialization
    - Still missing: the same native backward traversal support exposed more broadly for every path-pattern workflow
[~] Property-chain / role-composition reasoning
    - Implemented via targeted role saturation in `src/ontology_parse.py`
    - Also implemented natively for direct query semantics by compiling relevant chain conclusions into nested DAG restrictions
    - Current scope: reusable role consequences still flow through the targeted saturation path, while direct chain matching can avoid materialization
    - Still missing: broader benchmarking, oracle validation wiring, and more aggressive optimization for longer chains
[~] Transitive properties
    - Implemented via targeted role saturation in `src/ontology_parse.py`
    - Also implemented natively for direct existential query semantics via `EXISTS_TRANSITIVE_RESTRICTION` in `src/dag_eval.py`

Recommended implementation order
1. Add targeted native role saturation.
   - `subPropertyOf`
   - inverse properties
   - transitive properties
   - role chains
   - Restrict saturation to the property closure relevant to the queried DAG
2. Add DAG relevance-closure materialization.
   - Materialize only facts that can affect the queried class / DAG
   - Reuse this dependency analysis for future rule and saturation passes
3. Expand fuzzy semantics.
   - `sim_prop`
   - General continuous scoring beyond path chains
   - Strict-prefix mode if wanted
4. Strengthen validation.
   - Randomized supported-fragment generation
   - Compare against `owlrl` / `owlready2`
   - Check the local consistency-preservation guarantee with randomized cases and bucket failures by construct mix
   - Document expected mismatch cases for open-world vs local DAG semantics
5. Revisit the OWL EL track as a separate reasoning mode.
   - EL normalization pass
   - Semi-naive completion rules
   - Role saturation and relevance-aware saturation for EL entailments
6. Add the epistemic-bit branch as a separate execution mode.
   - Keep it separate until exact + fuzzy reasoning over real ontologies is stable
7. Benchmark on larger real datasets.
   - Then decide whether harder constructs are worth the architectural cost

Working summary of where the project stands today
- The repo has the core synthetic GPU-friendly DAG engine working.
- It now parses and reasons over real RDF / ontology inputs via `src/ontology_parse.py`, with separate schema/data loading.
- It supports a useful minimal OWL-like fragment including existential, universal, negation, direct disjointness, hierarchy propagation, and Horn-style class materialization.
- It has end-to-end examples plus an oracle-comparison harness against `owlrl` and `owlready2`.
- The next native-direction work is targeted role saturation and relevance-closure materialization.
- A more explicit OWL EL completion mode is plausible, but should be treated as a separate reasoning track rather than forced into every DAG query.
